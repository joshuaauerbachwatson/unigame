What is %appName%?

%generalDescription%

Planning a Game

%appName% does not provide "social" functions for finding like-minded players. Rather, players who want to play together must communicate ahead of time and agree on a few things.

  • They must all install this app.
  • They must agree on one of them to be "leader". The leader is responsible for specifying the number of players. Once all players have assembled, the leader is responsible for game setup, if any is required, and either takes the first turn or passes it to the next player. After that, the leader is just like any other player.
  • They must agree on whether they will play in "Nearby Only" mode. In this mode, users must be in proximity to each other (in bluetooth range or on the same wifi hub). This mode allows the Login step to be bypassed. If "Nearby Only" is not chosen, players can be anywhere on the internet but must identify themselves via a Login step.
  • They must agree on a "game token" that they will all use to identify the group. This can be any string of upper and lowercase letters, digits, hyphens and underscores, at least six characters long. The game token can be meaningful and need not adhere to "password strength" rules but it is better if it is not too likely to be chosen by other groups. The game token is what keeps different groups of players from merging into one bigger (unintended) group.


Solitaire Mode

If you identify yourself as leader and set the number of players to one, you are in "solitaire mode". No game token is needed and you can proceed from the player-control view to the game-playing views without further fuss. Solitaire mode is a good way to familiarize yourself with the overall look and feel of the game, and I recommend you use it for that purpose before playing real games with other players. However, %appName% is not intended primarily as a solitaire game.


The Top Controls

Most screens shown by %appName% have the following material at the top.

The Players row contains labels for all the players who are in the game. You are represented by the label You. Other players will show up as they join the game. If the expected number of players is known, there may be player slots labelled as "missing" or "searching".

A player label with a running figure in it is the player whose turn it is. The other players will have standing figures.

Below the Players row is a space where the latest chat message is shown. A change to the contents of this area will serve as notice that another player has something to say. In the picture, no chat has occurred yet.

Below the chat message area is a row with three buttons.

  1. The Chat button opens the chat view, which provides a scrollable complete transcript of the chat messages since you joined the game. A text entry area in this view allows you to send messages. The Chat button is disabled until there are at least two players in the game.
  2. The End Game button ends the game. It is disabled until you join the game. Once it is enabled it remained so until you either touch it to end the game or some other player ends the game or the game ends due to some unrecoverable error. Any player can end the game pre-emptively. It is good etiquette to let others know first (using the Chat button or by some other means). But, other players will find out quickly because they will get a pop-up explaining that you have left the game and game is over.
  3. The Help button brings up this help view. That button is always enabled.


Assembling Players

When you first open the app, the center of the screen will be occupied by the player-control view, which looks like the following.

Where you see I am: Logan Lester in the picture, that is the name by which you will be known to the other players. It will appear in chat messages sent by you and in the player labels in the other players' views. The first time you play, a randomly chosen name will appear there. You can touch and then overtype this name to identify yourself using a name more to your liking. That change will be remembered until you change it again.

The Leader toggle allows you to assert that you are the Leader for the game. As previously noted, only one player in the game should have this toggle on. The others should have it off.

The stepper for choosing the number o players will appear only if the Leader toggle is on. This is how the leader sets the number of players. It is also how you choose solitaire mode: by setting the number of players to one.

The Nearby Only toggle must be set the same way by all the players. Set it on if the players will be in close proximity and you wish to avoid the step of logging into the server.

The Game token: field is empty in the picture because no game token has been entered. Every player in the game must be using the same game token. To enter a new game token touch the Add button. Once you have entered a game token, it will appear in the Game token: field every time until you delete it or enter a new one. When you have used more than one game token without deleting, you can touch Saved Game Tokens to change which one is active.

The portion of the view devoted to game tokens will disappear in solitaire mode. You will not need a game token if you are not connecting to other players.

The two buttons Login and Join are shown at the bottom, unless you have chosen solitaire mode, in which case there will be a single Play button. If the game is Nearby Only, Login will be disabled, but, otherwise we require a login at least once a week. If Login is enabled and Join disabled then a fresh login is required as explained in The Login Process.

In addition to the login requirement, the Join button is enabled only if

  • you entered a game token
  • you have not yet joined the game

If the Join button is enabled, press it to start the process of joining the game (searching for the other players). Once every player has pressed this button (assuming they entered the other information correctly) the game should assemble in just a few seconds and all the players will have a consistent set of player labels.

Once you join the game, the player-control view may continue to show, as long as the Players area indicates that not all players are yet present. You can press End Game to cancel your membership in the game. As soon as all players are assembled, %appName% will shift to its next view. The top controls are the same in all views but the player-control area will be replaced as discussed in the later sections of this help.


The Login Process

%appName% uses a server to mediate traffic between players (except in Nearby Only mode). To protect the server from abuse by hackers and such, we require a login.

%appName% uses Auth0 (a respected online secure login provider) to authenticate its users. When you touch Login you are first told that you are about to share information with auth0.com. Assuming you agree, you are then presented with a dialog crafted by Auth0, giving you the choice of several different ways of authenticating yourself. You can create an account using your email address and choosing a password (which you will then use for subsequent logins). Or, you can log in using existing accounts you may have at GitHub, Google, or Apple. We may enable additional account providers in the future (Auth0 supports many of these). Using a service with which you already have an account avoids the need to remember yet another password and reduces the amount of information about you that Auth0 needs to store directly.

Once you are logged in, the data area on your device will contain an access token which expires in about one week. This means you will need to login once a week or so. The access token is the only information about you that %appName% stores on your device. The server does not store any information about you. Auth0 stores enough information to figure out who you are from the access token you use. So, %appName% can find out, from Auth0, what account you used to identify yourself. It does not routinely do so, and it does not make routine use of this information, nor does it ever share it with a third party. The capability is there in case some %appName% players turn out to be bad actors with malicious intent who need to be blocked from using the server.


Using the Integrated Chat Channel

%appName% has an integrated chat chennel for the convenience of the players. You are also free to use means outside the app to communicate with other players. For example, in Nearby Only you might well be in the same room. Or, you could use telephone or iMessage to communicate even if you are remote from each other.

Chat is only available when the game has more than one player. Your chat transcript begins when you join the game and becomes unavailable when the game ends, so it is not a long-term record of what was said. There is no centralized transcript, so a player joining the game late does not see chat messages that occurred earlier.

To initiate a chat, touch the Chat button near the top left of the screen. The resulting view looks like like the following. The entire transcript of messages should show in the body of the screen, oldest first. If there are more messages than will fit in the view, it becomes scrollable with the most recent messages showing. The text entry area above the transcript will receive focus and a keyboard should remain open so that you can type your own messages. A message is sent every time you touch the Return key.

When you are done chatting, press the navigation back-arrow to return to the previous view. On any view, the last chat message in the transcript will always be displayed, along with the time it was posted.

Reporting Problems

At the top of this help text is a link for reporting problems. Touching it will bring up an initialized email sending view with the recipient set to the %appName% reporting address and some instructions in the body of the message (which you should delete and replace with your specific problem report).

You can use this capabiity to send any kind of commentary or input that you like, but a key purpose is to support reporting bugs and other problems that you are having with the app. To this end, the email, by default, contains up to two (often just one) logs of recent activity. When you are reporting a serious problem, such as a crash or misbehavior of some sort, it is a good idea to leave these logs in place since they help in diagnosing problems. They include only information about recent sequences of actions inside the app itself and do not include any personal information. However, if you feel more comfortable removing these logs you are free to do so.
